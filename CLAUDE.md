This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run dev` - Start development server (Express + Vite HMR on port 5000)
- `npm run build` - Production build (Vite for client, esbuild for server)
- `npm run start` - Start production server from `dist/`
- `npm run check` - TypeScript type checking
- `npm run db:push` - Push Drizzle schema changes to PostgreSQL (requires `DATABASE_URL`)

No test runner or linter is configured.

## Architecture

Full-stack TypeScript monorepo with three main directories:

- **`client/`** - React SPA (Vite, Wouter routing, TanStack Query, Tailwind + shadcn/ui)
- **`server/`** - Express REST API with Passport authentication
- **`shared/`** - Drizzle schema definitions and Zod validators used by both sides

### Path Aliases

Configured in both `tsconfig.json` and `vite.config.ts`:
- `@/*` → `client/src/*`
- `@shared/*` → `shared/*`

### Authentication

Passport local strategy with scrypt password hashing (`server/auth.ts`). Session-based auth using `express-session` with `memorystore`. Auth routes: `POST /api/register`, `POST /api/login`, `POST /api/logout`, `GET /api/user`. Most API routes are protected via `requireAuth` middleware. The frontend uses an `AuthProvider` context (`client/src/hooks/use-auth.tsx`) that gates all routes behind login.

### Data Flow

The server uses a storage interface pattern (`IStorage` in `server/storage.ts`). When `DATABASE_URL` is set, it uses `DatabaseStorage` (Drizzle ORM + Neon serverless PostgreSQL). Otherwise it falls back to `MemStorage` (in-memory Maps with seeded NYC activities).

Routes (`server/routes.ts`) validate request bodies using Zod schemas generated by `drizzle-zod` (`insertUserSchema`, `insertActivitySchema`, etc.) before passing to storage methods. Authenticated routes inject user IDs from `req.user` rather than accepting them from the client.

On the client, `TanStack Query` manages server state. API calls go through `apiRequest()` in `client/src/lib/queryClient.ts` which handles credentials and error checking.

### Database Schema

Five tables defined in `shared/schema.ts`: `users` (with username/password), `activities`, `userActivityInterests`, `matches`, `messages`. All use UUID primary keys. Matching is activity-based: users express interest in activities, and potential matches are discovered through shared activity interests.

### Frontend Routing

Wouter routes defined in `client/src/App.tsx`. Unauthenticated users see the auth page. Authenticated routes: `/` (discover), `/profile-setup`, `/matches`, `/messages`, `/conversation/:matchId`, `/create-activity`. Bottom navigation bar provides primary navigation.

### Dev Server Integration

In development, Express serves as both the API server and Vite's HMR host via `server/vite.ts`. In production, Express serves the built static files from `dist/public`.
